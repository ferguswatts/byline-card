# Byline Card

**Know who's writing what you're reading.**

A Chrome extension and open data project that brings transparency to New Zealand political journalism. Hover over any journalist's byline on a NZ news site and see their coverage distribution, documented connections, and background — all sourced, all verifiable.

## Why this exists

Every journalist has a perspective. That's normal and human — it's what makes journalism interesting. But readers deserve to know what that perspective looks like across hundreds of articles, not just the one in front of them.

Byline Card doesn't label journalists as "biased" or "neutral." Instead, it shows the **distribution** of their coverage — how many articles lean left, centre, or right — and lets you draw your own conclusions. A journalist who writes 90% centre articles and 10% centre-left is different from one who writes 50/50, and both are different from one who writes opinion pieces that consistently lean one direction. The distribution tells the story.

We believe transparency improves trust in media. When readers can see a journalist's track record before reading an article, they're better equipped to identify framing, weigh arguments, and form their own views. That's not anti-media — it's pro-reader.

Every connection on a card links to a public source. Every score is generated by an open, documented methodology. The code is open source. If we're asking for transparency from journalists, we should hold ourselves to the same standard.

## What it shows

When you hover over a journalist's byline on a supported NZ news site, a card appears with:

- **Coverage distribution** — percentage of articles scoring Left / Centre-Left / Centre / Centre-Right / Right, based on AI analysis of framing, source selection, language, and editorial choices
- **Documented connections** — family ties to politicians, marriages to other journalists, employer relationships. Every connection links to its public source
- **Background** — biographical context, career history, notable facts
- **Confidence level** — based on how many articles have been scored (more articles = higher confidence)

## How scoring works

Each article is scored on a **-1.0 to +1.0 scale** across five dimensions:

| Dimension | What it measures |
|---|---|
| **Framing** | How the topic is presented; who is protagonist vs antagonist |
| **Source selection** | Which voices are quoted; are opposing views included |
| **Language** | Loaded or emotive language ("slammed", "radical", "controversial") |
| **Topic emphasis** | What aspects are highlighted vs downplayed |
| **Omission** | What relevant context or perspectives are missing |

A critical principle is **"cui bono"** — who benefits from this story being published? An article exposing a right-wing party's internal divisions benefits the left, even if the prose reads as balanced. The editorial choice of what to write about is itself a signal.

### Score buckets

| Bucket | Score range |
|---|---|
| Left | below -0.6 |
| Centre-Left | -0.6 to -0.2 |
| Centre | -0.2 to +0.2 |
| Centre-Right | +0.2 to +0.6 |
| Right | above +0.6 |

The card shows the **distribution** across all scored articles — not a single score. A journalist may write centrist news pieces and left-leaning opinion columns. The distribution shows both.

### Calibration

The scoring prompt was calibrated against human-scored ground truth (15 articles scored by a human with NZ political expertise). The AI scores achieve a Pearson correlation of **r = 0.74** with human judgment. The prompt was iteratively tuned — verbose instructions actually degraded performance, while a concise "cui bono" addition improved it.

## Architecture

```
Chrome Extension (WXT + TypeScript)
  └── Content script detects bylines, renders hover card (Shadow DOM)
  └── Loads data.json from GitHub, falls back to bundled copy

data.json ← exported by pipeline after each scoring run

Python Pipeline (SQLite + async)
  ├── 7 per-site adapters (NZ Herald, Stuff, RNZ, 1News, Newstalk ZB, Spinoff, Newsroom)
  ├── URL discovery via sitemaps (1.4M+ article URLs catalogued)
  ├── Author extraction and journalist tagging
  ├── Claude API scores each article (-1.0 to +1.0)
  ├── Aggregator computes distribution percentages
  └── Exporter writes data.json + generates static dashboard

Live dashboard hosted on GitHub Pages
  └── Self-contained HTML — all data inline, no server needed
```

Two moving parts. No API server. No database server. No hosting costs.

## Pipeline

The scoring pipeline discovers articles across all supported outlets, extracts text, scores each article via LLM, and exports the results.

### Setup

```bash
cd byline-card
python -m venv venv
source venv/bin/activate
pip install -r pipeline/requirements.txt
playwright install chromium
cp .env.example .env
# Add your API key(s) to .env
```

### Run

```bash
# Normal run — score new articles for all journalists
python -m pipeline.run

# Backfill — fetch up to 200 articles per journalist (historical)
python -m pipeline.run --backfill

# Discover article URLs from sitemaps (all outlets)
python -m pipeline.discover_urls

# Tag discovered URLs with author names
python pipeline/tag_all_authors.py

# Generate the dashboard
python -m pipeline.dashboard --open

# Re-score articles after prompt changes
python -m pipeline.run --rescore
```

### Automated pipeline

A GitHub Actions workflow runs daily, scoring new articles and committing updated `data.json` to the repo. The extension picks up changes automatically.

## Data

### Journalists

38 NZ political journalists currently tracked across 8 outlets. Defined in `data/journalists.csv`.

### Connections

Documented relationships (family, spouse, employer) with source URLs. Defined in `data/connections.csv`. Every connection must have a verifiable public source — no unsourced claims.

### Article discovery

Over 1.4 million article URLs catalogued from sitemaps across all outlets. Author names extracted via:
- **Stuff**: JSON API (no HTML fetch needed)
- **Newstalk ZB**: Author parsed from URL structure
- **NZ Herald / RNZ / Spinoff**: HTML metadata extraction (trafilatura + JSON-LD)

## Contributing

We welcome contributions:

- **Add a journalist** — edit `data/journalists.csv`
- **Add a connection** — edit `data/connections.csv` (source URL required)
- **Add a site adapter** — create `pipeline/sites/newsite.py` implementing `SiteAdapter`
- **Fix an error** — open an issue or PR with corrected information
- **Improve scoring** — the calibration test suite (`tests/pipeline/test_calibration.py`) validates prompt changes against human ground truth

All contributions are reviewed before merging. Connections require public source URLs.

## Project principles

1. **Transparency cuts both ways** — if we ask for transparency from journalists, our methodology, code, and data must be open too
2. **Distribution over labels** — we show the spread of coverage, not a single "bias score"
3. **Sources or it didn't happen** — every connection links to a public source. Every score links to the methodology
4. **Bias is human** — having a perspective isn't a flaw. But readers deserve to see it before it shapes their thinking
5. **Privacy-first** — the extension makes no API calls, collects no user data, and runs entirely from bundled data
